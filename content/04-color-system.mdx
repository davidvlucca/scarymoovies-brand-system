# Color System

ScaryMoovies uses four precisely defined color scales — Purple, Red, Black, and White — each with ten steps. Every color decision in the interface maps back to one of these scales through a layer of semantic tokens.

## Philosophy

The system is dark-first by design — not as a style choice, but as a brand statement. Black is the natural environment of horror. Purple carries the weight of something otherworldly, the color of bruises and shadows cast by things that shouldn't exist. Red is blood, used sparingly so it means something when it appears. White is pure contrast — legibility against the void.

## Purple — Primary Brand Color

Purple drives every interactive moment on the platform. It marks what can be clicked, what is active, and what deserves attention. Use it for interactive elements only — buttons, links, focus rings, active navigation states.

## Red — Horror Accent

Red is reserved. It appears only for danger states, destructive actions, and the specific horror accents that make this platform feel like ScaryMoovies rather than any other dark UI. Overuse destroys its power.

## Black — Surface & Background

The black scale defines every surface and background in the interface. It is not a single flat black — it is a family of near-identical dark values that create subtle depth without introducing obvious contrast breaks.

## White — Text & Contrast

White handles all text and light-on-dark contrast. The scale from `white-50` through `white-900` creates the typographic hierarchy — primary text, secondary labels, muted annotations — all without introducing any actual color.

## Semantic Tokens

Raw scale values like `purple-400` are never referenced directly in production code. Instead, every UI decision uses a semantic token — a named alias that describes the token's *role*, not its value. This decouples design decisions from implementation: the palette can evolve without touching component code.

## Usage Rules

- **Purple is for interactive elements only.** Decorative use of purple dilutes its signal value as an affordance indicator.
- **Red is danger and horror accent only.** Never use it for decorative purposes, category colors, or hover states unrelated to destructive actions.
- **Never use purple-200 or lighter as text on dark backgrounds.** These steps fail WCAG AA contrast requirements at body text sizes.
- **Always reference semantic tokens in production code**, not raw scale values. Write `var(--accent-primary)`, not `var(--purple-400)`.
- **Do not darken `--border-subtle`.** Its 0.06 alpha opacity is intentional — it creates structure without competing with content.
- **Text hierarchy is white-only.** Do not use purple or red for headings or labels outside of specific badge contexts.

## Accessibility

WCAG 2.1 contrast ratios for key pairings against `--bg-primary` (`#0c0a0e`):

| Foreground | Token | Hex | Ratio | Level |
|---|---|---|---|---|
| Primary text | `--text-primary` | `#fefefe` | 19.5 : 1 | AAA |
| Secondary text | `--text-secondary` | `#adacae` | 7.2 : 1 | AAA |
| Accent hover | `--accent-hover` | `#785492` | 4.6 : 1 | AA |
| Accent primary | `--accent-primary` | `#5d337d` | 3.8 : 1 | Fails AA† |
| Danger | `--accent-danger` | `#751111` | 3.5 : 1 | Fails AA† |

†`--accent-primary` and `--accent-danger` do not meet WCAG AA for small body text (4.5 : 1 required). They are used exclusively for interactive affordances and icon contexts where large size or bold weight compensates. Never use them for paragraph text.
